### Methodology

For each contract-level alert (ticker + strike + expiration + option_type + datetime):

```sql
SELECT * FROM flow_alerts
WHERE ticker=? AND strike=? AND expiration=? AND option_type=?
  AND ts BETWEEN <30 min before post> AND <3 hours after post>
ORDER BY ts;
```

- **CAUGHT** = at least one matching row with conviction in {MEDIUM, HIGH, SWEEP} AND sentiment matching call→BULLISH / put→BEARISH
- **WRONG-SIDE** = MEDIUM/HIGH row exists but sentiment opposite
- **WRONG-SIDE-LOW** = LOW conviction + wrong sentiment
- **PARTIAL** = LOW conviction only (we saw it but filter killed it)
- **OUT-OF-CHAIN** = ticker scanned in window but no rows for that strike+exp
- **MISSED-NO-TICKER** = ticker in universe but no flow_alerts rows whatsoever in the window
- **OUT-OF-UNIVERSE** = ticker not in `server.tickers.all_tickers()`

## Pattern analysis: where we systematically fail

### 1. **Side detection on calls bought at MID-of-spread** — biggest single bug

When premium is paid above the midpoint but below the ask (e.g. avg fill 89.45 on 92.0×95.35 spread for MU 1020C 9/18), our `side` classifier tags MID → NEUTRAL. FL0WG0D's Bullflow tagging shows the same fill as an ASK-side aggregator with ASK_SIZE >> BID_SIZE. The 5/13 17:34 flip on QCOM 270C from MID/NEUTRAL to BID/BEARISH illustrates the same problem in reverse — the spread shifted and we re-tagged a passive trader instead of recognizing the BULLISH delta-positive run.

**Affected contracts: 5 of 41 (MU 1020 5/12, MU 1020 5/12 sweep, SPY 745, QCOM 270, AVGO 447.5).** Hit rate would jump from 39% → 57% if we tagged side correctly on these.

Recommended fix:
- When `vol_oi > 1.5` and `last_price > (bid + ask) / 2` AND ask premium > $1M, override MID → ASK regardless of strict midpoint rule.
- Better long-term: integrate condition=95 (ISO sweep) from OPRA — these aggregated multi-leg fills are exactly what Bullflow's "Ask:Bid ratio" UI surfaces.

### 2. **Chain coverage gaps for non-standard strikes** (6 of 41)

We caught MU 1020 and 1570 but not 1030, 1500, 1550 (only every $50 visible). Same for AA — K=66 and 63 in chain, but FL0WG0D's call was 70. Same FCEL 30 strike — we only have 19/25.

The chain selection logic in `_select_expirations` and `_select_strikes` works for "near ATM" but fails for the deep-OTM tail FL0WG0D specifically targets — Bullflow biases toward the strikes where premium concentration is highest *because that's where insiders trade*, and that's often well above our default strike coverage.

Recommended fix:
- For tickers with `flow_alerts` showing $1M+ notional in any strike, expand the chain pull to ±50% around spot for the same expiration on the next scan cycle.
- For known parabolic names (memory criterion: high realized vol last 5 days), pre-fetch the top 30 OI strikes regardless of moneyness.

### 3. **Ticker-not-scanned gaps in universe** (4 of 41)

PYPL, FCEL, AA(5/12), STX(?) all have rows in the DB historically but ZERO rows in the FL0WG0D-relevant window. PYPL hasn't been refreshed since 5/5. AA only has rows on 5/11, none 5/12. **421 universe tickers but only 165 distinct tickers in today's flow_alerts.** That's 61% of the universe sitting idle.

This is a scanner-freshness bug not a detector bug. Likely root cause: the scan cadence has degraded as we added more tickers and the per-cycle budget runs out of time. Likely worth profiling `flow_alerts.py` for per-ticker latency.

## Recommendations — ranked by ROI

### P0 — Side detection on MID-of-spread fills

Single biggest delta. ~5 of 11 wrong-sided cases are this pattern. The fix is local to `flow_alerts.py` side-classification — should be ~30 minutes of work + a backtest replay on 5/12 to validate.

**Pseudo-code:**
```python
def classify_side(last, bid, ask, vol, oi):
    # Existing: ASK if last >= ask*0.95, BID if last <= bid*1.05, else MID
    mid = (bid + ask) / 2
    if last >= ask * 0.95:
        return "ASK"
    if last <= bid * 1.05:
        return "BID"
    # NEW: above-mid + heavy volume + Vol/OI surge → still aggressive ASK lean
    spread = ask - bid
    if spread > 0 and last >= mid + spread * 0.25 and (vol / max(oi, 1)) > 1.5:
        return "ASK"
    if spread > 0 and last <= mid - spread * 0.25 and (vol / max(oi, 1)) > 1.5:
        return "BID"
    return "MID"
```

This is essentially a "leaning toward the offer" heuristic that mimics what Bullflow's UI shows in the Ask Size / Bid Size / Mid Size breakdown.

### P1 — Scanner freshness (165/421 ticker coverage in 24h)

61% of universe is stale. Profile `flow_alerts.py` to find where time is spent. The likely fix is either:
- Increase scan cycle parallelism (ThetaData should handle 10+ concurrent option_history_quote calls).
- Tier the universe: hot-list (top 50 by avg vol_oi over last 5d) scanned every 5 min, warm list every 15 min, cold list every 60 min.
- Drop dead tickers — anything that hasn't produced a MEDIUM+ row in 30 days drops from active scan.

### P2 — Chain expansion on parabolic names

When ANY strike of a ticker shows $1M+ notional in a single scan cycle, the next cycle pulls ±50% spot strikes for the same expiration. This would have caught MU 1030 and AA 70 in this audit.

### P3 — Universe additions

Add **TTD, TOST, NXE, KEEL, SSRM, NPWR, LUMN, SONY, VNET, QURE** (10 tickers, +2.4% to universe). Each has thematic concentration justifying inclusion. FRMI/RR I'd need to look up first before adding.

### P4 — Bullflow-parity OCR / vision pipeline (optional)

For tier-1 flow accounts (FL0WG0D, FlowbyBobby, etc.), build a daily X scrape → Claude-vision-OCR → flow_alerts cross-check → daily "What did they catch that we didn't" Telegram digest. This audit took ~3 hours of manual work; automated it's a 10-minute cron job and gives us continuous external validation.

### P5 — Heatmap-style SPY/SPX context (intentionally out of scope)

FL0WG0D's commentary posts ("7385 hold then 7400 target") aren't something we're trying to replicate — that's manual macro layer, not flow detection. Keep separate.

## Honest limitations of this audit

1. **2.2 days, not 7.** The brief asked for 7 days; X timeline rendering + my manual scroll budget capped it earlier. The patterns are real but the count is small (n=41). A second pass with chain backfill from 5/6-5/10 would strengthen the chain-coverage finding in particular.

2. **Strikes/exps extracted by eye** from Bullflow chart images during the live scroll, not via systematic vision OCR. There's a small risk of mis-read on edge cases (e.g. did I read "$0.88" as $0.88 or 0.88 fill — both plausible).

3. **Some "OUT-OF-CHAIN" verdicts may collapse to "MISSED-NO-TICKER" or vice versa** depending on exact time window — I used 30 min before / 3 hours after his post, but his Bullflow charts often back-reference fills from hours earlier. A larger window would shift a couple of cases.

4. **"WRONG-SIDE" verdicts may be correct in aggregate.** Some plays (QCOM 270C on 5/12 ASK / 5/13 BID flip) are arguably "right side at the time of original fill, wrong side later" — Bullflow shows a single screenshot; we have a time series. If you look at our 5/12 19:04 row on QCOM 270, we tagged MID/NEUTRAL but his post timestamp was hours later, when the contract was already paying retail exit fills.

5. **Caught/wrong-side asymmetry on basket aggregates.** Posts like "$1.3 million into these $AAPL calls" don't specify strike/exp; I only verified ticker-window presence. The "CAUGHT" verdict on these (AAPL, MBLY, NOK, ON, HPE) is a weak verdict — we definitely saw something on the ticker but we didn't necessarily catch the specific contract he meant.

## Net judgment

**The headline hit rate is genuinely 39%** of in-universe contracts caught with right side + meaningful conviction. The remaining 61% break down evenly between three causes — side-detection (~15-20%), chain coverage (~20%), and scanner staleness (~20%). The first two are local fixes worth a few hours each. The third is architectural and probably a few days of work. The 32% out-of-universe rate is the easiest delta — adding 10 tickers tomorrow lifts overall caught rate from 27% to ~32% by simple denominator math.

If the user asked "is our scanner Bullflow-parity?" the honest answer is no — but it's not catastrophically far off either. The hard part (detecting unusual single-contract concentration on the day-of) we do correctly on the cases where (a) the ticker is scanned, (b) the strike is in our chain, and (c) the spread is wide enough that side detection doesn't get confused. Fix those three things and 39% → ~65-70% is realistic.

The MU 1020 / 1570 result is the bull case: we DID catch the most consequential trade of the week (the $6.4M call from 5/12 that's a key node in the user's substack thread). We just tagged it MEDIUM/MID/NEUTRAL when it deserved HIGH/ASK/BULLISH. Bug #1 of the "side detection" P0 fix above.
